# Session 11: Test the Move
*"Did the post-move build actually get beyond the wall?"*

## What This Session Covers

Session 11 uses peer testing to check the constraint-and-move story students are already documenting in the end-of-course assignments.

This is not generic polish. A classmate should be able to see:

- what Space 2 could do,
- what Space 2 could not do,
- what move Space 3 made,
- whether the move helped,
- and what the move cost.

The end-of-course assignments in Google Classroom remain the guide. Tonight gives students evidence and feedback they can use for the Space 3, Research Journal, Research Paper, Demo Day Presentation, and GitHub Profile assignments.

## The Core Teaching Line

> Peer feedback is not just "do you like my Space?" It is evidence about whether the move actually changed what the project can do.

## Session Flow

### Move 1 - Reopen the Assignment Arc

Start with the eight end-of-course assignments. The focus tonight is:

- **Space 2 - The Rudimentary Build**
- **Space 3 - The Post-Move Build**
- **Research Paper**
- **Research Journal**
- **Demo Day Presentation**

Students should have a wall sentence and a move sentence ready, even if the build is still partial.

### Move 2 - Explain the Wall and Move to a Peer

Each student explains their project to a classmate in four sentences:

```markdown
Space 2 can...
Space 2 cannot...
The move in Space 3 is...
The tradeoff is...
```

If they cannot say this clearly, that is the first thing to fix.

### Move 3 - Run Before/After Tests

When both Spaces are available, peers test the same input in Space 2 and Space 3.

When only Space 3 is available, peers test whether it does the thing Space 2 could not do.

When Space 3 is broken, peers help read the error, identify the wall, and suggest the next move.

The feedback form asks for what changed, what still failed, and what tradeoff is visible.

### Move 4 - Revise the Story

Students use peer feedback to revise:

- the Space 3 README section explaining the constraint and move,
- the Week 10 or Week 11 journal entry,
- Paper sections 3-6,
- the Demo Day presentation outline.

This is where the work becomes clearer. Not necessarily more polished - clearer.

### Move 5 - Build the Demo Day Skeleton

Before leaving, students draft the five-part presentation from the assignment:

1. What I wanted to build
2. The constraint
3. The move
4. What it cost
5. What is next

They should know which tab, screenshot, Space output, or error message they will show for each part.

## What Students Should Leave With

- peer feedback on whether the move helped
- at least one before/after result, or one documented blocked-status result
- one tradeoff stated clearly
- a revised or planned Space 3 README note
- a Demo Day presentation skeleton
- a clear list of which end-of-course assignments still need updating
